# Replay Workbench 缺数据问题专项解决方案

## 1. 文档目的

这份文档不是泛泛讨论“实时链路”或“补数能力”，而是**专门解决当前最痛的问题**：

> 用户打开 Replay Workbench 时，经常发现图表缺数据、只有一小段、黑屏、右侧有控件但主图无内容；即使当前休市，系统也没有给出可信解释、补救路径或可恢复的历史视图。

你指出的问题非常关键：

- 现在 UI 再怎么调整，如果**数据面不稳**，页面仍然不可用
- 特别是**休市时**，实时流本来就弱，但系统也没有自动切换到“历史可用模式”
- 用户看不到：
  - 当前是没数据
  - 还是正在补数
  - 还是休市导致没有最新流
  - 还是缓存可恢复但没用上
  - 还是 ATAS 侧根本没把当前窗口的 bars 送过来

所以本专项文档的目标是：

> 把“缺数据”问题拆清楚，建立一套**可判断、可补偿、可降级、可解释、可恢复**的完整方案。

---

## 2. 问题定义

当前“缺数据”不是一种单一故障，而是至少包含以下 6 类情况：

### 2.1 首屏返回过少
例如：
- 1m 图只返回了几十根 bar
- 页面看起来像“只剩一小截”
- 用户以为系统坏了

本质：
- 可能是首屏窗口裁剪过小
- 但页面没有明确告诉用户“当前仅加载首屏窗口，后台正在扩窗”

### 2.2 中间缺段
例如：
- 10:21 到 10:27 缺了一段
- 图表时间轴连续性不稳定
- 数据显示短一截

本质：
- 历史 bars 没送全
- live tail 没补上
- backfill 没真正完成

### 2.3 尾部停更
例如：
- 最后一根 bar 不再刷新
- 图表停在很早之前
- 用户不确定是休市还是断流

本质：
- 实时 continuous-state 断了
- 或当前市场真的休市
- 但系统没有做状态分类

### 2.4 当前窗口完全无数据
例如：
- 图表主区域黑着
- 右侧面板有内容，但 K 线为空
- 用户完全不知道为什么空

本质可能是：
- 当前 symbol/timeframe/window 组合没有命中缓存
- 后端 snapshot 构建失败
- 历史数据确实不存在
- 数据存在但前端 merge/render 没成功

### 2.5 休市时“看起来像故障”
例如：
- 当前是周末 / 维护时段 / 非交易时段
- 没有最新 bar 是正常的
- 但系统既没显示“休市”，也没自动回退到最近可用历史

这会让用户误以为：
- 页面坏了
- 服务掉了
- 补数没做

### 2.6 缓存明明可用，但没有拿来恢复
例如：
- 上次明明看过该品种/周期
- 本地或后端应该有缓存
- 但新开页面还是从空白开始

这属于非常影响体验的问题。

---

## 3. 为什么“休市时缺数据”会特别严重

休市时，问题会被放大，不是因为系统一定更坏，而是因为：

### 3.1 没有实时流来掩盖问题
开市时即使历史有点缺，continuous-state 还能不断推新尾巴，用户感觉“图还活着”。

休市时：
- 没有最新成交
- 没有新的 continuous-state 修正尾部
- 缺口就完全暴露出来

### 3.2 用户更依赖历史完整性
休市时用户主要做的是：
- 复盘
- 看过去 1 天 / 3 天 / 7 天结构
- 做计划

所以系统此时必须：
- 历史完整
- 或至少告诉用户哪些地方不完整

### 3.3 “没有实时”不等于“没有图”
这是当前最大设计误区之一。

正确逻辑应是：

- 休市 = 可能没有新的尾部 bar
- 但**不代表当前页面应该无图**
- 用户仍应看到：
  - 最近完整历史
  - 最近收盘位置
  - 最近一次同步时间
  - 当前休市状态
  - 缺口状态

如果休市时页面几乎空白，那就是系统设计有问题，不是市场的错。

---

## 4. 缺数据问题的目标状态

目标不是“尽量少出错”，而是：

### 4.1 无论开市还是休市，页面都必须尽量可用
最少也要做到：

1. 有缓存时先恢复缓存
2. 有历史时先显示历史
3. 无实时时明确标记休市/历史模式
4. 有缺口时明确标记“正在补数 / 尚有缺口”
5. 如果真的什么都没有，也要给出明确原因和下一步动作

### 4.2 用户必须一眼知道当前数据可信度
页面需要能明确表达：

- 当前是实时图还是历史图
- 最近同步时间
- 是否仍有缺口
- 是否处于补数中
- 当前看到的是缓存恢复结果还是后端最新结果

### 4.3 首屏优先可见，不允许长时间空白等待
就算后台还在：
- 扩窗
- 补数
- 合并

前端也应该优先展示：
- 最近缓存
- 最近可用快照
- 历史回放窗口

而不是让主图区黑着等。

---

## 5. 核心设计原则

### 5.1 缺数据不是异常分支，而是系统常态之一
必须把“缺数据”当成系统设计的一等公民。

因为实际交易环境里，缺数据会持续出现：
- 网络抖动
- ATAS 重启
- 非交易时段
- 历史同步不及时
- 某个窗口没命中缓存

### 5.2 数据状态必须是显式状态，不是靠用户猜
系统必须显式区分：

- `live_complete`
- `live_degraded`
- `historical_complete`
- `historical_incomplete`
- `backfilling`
- `market_closed_cached`
- `market_closed_no_cache`
- `empty_window`
- `snapshot_failed`

### 5.3 显示层优先保证“可解释”
如果当前没有最新流，也必须告诉用户：

- 为什么没有
- 当前还能看什么
- 系统正在做什么
- 要不要手动触发什么

### 5.4 恢复优先级必须固定
任何时候图表数据恢复顺序必须明确：

```text
本地缓存 > 后端工作台缓存 > 后端 snapshot > backfill 后新 snapshot > live tail
```

说明：
- 先让用户看到“最近可用画面”
- 再静默追平到更高版本

### 5.5 休市时系统应自动切到“历史可用优先模式”
也就是说：
- 不再等待实时流来证明页面活着
- 而是主动展示最近可用历史 + 状态说明

## 7. 系统必须能识别的 8 种页面状态

### 7.1 `live_complete`
定义：
- 市场开市
- 实时流正常
- 最近窗口完整或接近完整

前端显示：
- `实时`
- `完整`
- `最近同步 2 秒前`

### 7.2 `live_degraded`
定义：
- 开市
- 有实时，但出现缺口或延迟

前端显示：
- `实时（降级）`
- `缺失 23 根`
- `正在自动补数`

### 7.3 `historical_complete`
定义：
- 市场关闭或用户看的就是纯历史窗口
- 历史完整

前端显示：
- `历史`
- `完整`
- `最近可用 bar 于 ...`

### 7.4 `historical_incomplete`
定义：
- 当前没有实时流
- 但历史也不完整

前端显示：
- `历史（不完整）`
- `缺失 ...`
- `补数中`

### 7.5 `market_closed_cached`
定义：
- 当前休市
- 没有实时是正常的
- 但缓存可恢复，历史可看

前端显示：
- `休市`
- `已恢复最近缓存`
- `最后可用数据：...`

### 7.6 `market_closed_no_cache`
定义：
- 当前休市
- 没缓存
- 后端也没有足够历史快照

前端显示：
- `当前休市，且本地/后端均无可恢复缓存`
- 提供“查看最近可用交易日”“切换时间范围”“发起历史导入”入口

### 7.7 `empty_window`
定义：
- 当前时间窗口内确实没有任何数据
- 但不是系统故障

例如：
- 用户选了错误时间段
- 该 symbol/timeframe 在该窗口内确实无记录

前端显示：
- `当前窗口无数据`
- 给出最近有数据的时间段建议

### 7.8 `snapshot_failed`
定义：
- 后端快照构建失败或渲染失败

前端显示：
- `图表构建失败`
- 显示错误原因
- 提供重试 / 查看缓存 / 回退到最近成功快照

## 8. 休市场景的专项处理方案

这是本文件最关键的部分。

### 8.1 休市时不应依赖 live-tail 决定页面是否有图
当前很多系统会隐式地把“有没有新数据”当成“有没有图”的依据。

这是错的。

正确逻辑：

- 如果市场关闭，`stream_state = offline` 并不代表失败
- 此时优先使用：
  - 最近本地缓存
  - 后端工作台缓存
  - 最近成功 snapshot
  - 最近已同步历史 bars

### 8.2 休市状态识别
建议后端统一输出 `market_state`：

- `open`
- `premarket`
- `postmarket`
- `closed`
- `maintenance`
- `unknown`

来源可以结合：
- 交易时段表
- 当前时间
- 最近 bar 时间
- adapter 心跳情况

### 8.3 休市时的前端默认行为
如果 `market_state = closed`：

前端应自动：

1. 显示最近可用历史数据
2. 默认把状态切成“历史模式”
3. 在状态栏明确写：`当前休市，显示最近可用历史`
4. 如果有缺口，继续显示：`历史不完整，正在补数`
5. 不要因为没有新 bar 就把主图判为空

### 8.4 休市时的“最近可用历史回退”
当当前窗口为空时，系统应自动尝试：

#### 第一步：当前窗口内找最近可用 bar
如果没有，继续。

#### 第二步：回退到最近一个有数据的窗口
例如：
- 用户看最近 7 天
- 当前最后 12 小时无数据（休市）
- 但前一天 RTH 有完整历史

则系统仍应显示那部分历史，而不是黑屏。

#### 第三步：提示用户“当前已回退显示最近可用历史”
例如：
- `当前市场休市，已回退到最近可用交易时段数据（截至 03/20 21:00 UTC）。`

## 9. 缺数据检测机制

### 9.1 检测维度不能只有 missing_bar_count
缺数据检测至少要拆成：

1. `window_has_any_data`
2. `latest_bar_too_old`
3. `missing_bar_count`
4. `gap_segment_count`
5. `cache_available`
6. `last_successful_snapshot_at`
7. `market_state`
8. `stream_state`

### 9.2 需要新增的判断函数
后端建议新增：

- `detect_market_state(symbol, now_utc)`
- `detect_window_emptiness(candles, window_start, window_end)`
- `detect_latest_bar_staleness(candles, timeframe, market_state)`
- `compute_integrity_status(candles, timeframe, window_days)`
- `resolve_fallback_display_window(symbol, timeframe, requested_window)`
- `build_missing_data_status(...)`

### 9.3 休市时的 stale 判断不能和开市时一样
例如：
- 开市时 1m 图如果 3 分钟没更新，应视为可疑
- 休市时 1m 图 8 小时没更新，不应自动判故障

所以 stale 判断必须结合：
- market_state
- session calendar
- latest_available_bar_at

## 10. 数据恢复策略

### 10.1 恢复优先级
统一恢复顺序：

#### Level 1：本地缓存恢复
适合：
- 页面重开
- 新标签页打开
- 浏览器刷新后秒开

#### Level 2：后端工作台缓存恢复
适合：
- 本地缓存失效
- 跨页面恢复
- 新开页面快速拿最近状态

#### Level 3：后端 snapshot 重建
适合：
- 缓存不可用
- 需要更高版本历史

#### Level 4：自动补数后替换
适合：
- snapshot 不完整
- 有 gap
- 需要 history bars 回填

### 10.2 恢复时的 UI 行为
恢复必须分阶段可见：

#### 阶段 A：先显示缓存
状态条：
- `已恢复缓存`
- `正在校准最新数据`

#### 阶段 B：校准后端 snapshot
状态条：
- `已同步最新历史快照`

#### 阶段 C：补数中
状态条：
- `最近7天缺失 240 根，正在补数`

#### 阶段 D：补数完成
状态条：
- `历史完整性已恢复`

## 11. 补数策略专项

### 11.1 缺数据补偿不能只在开市时工作
补数必须在以下场景都可触发：

1. 开市中发现 gap
2. 收到 history ack 后仍不完整
3. 页面首次 build snapshot 就发现缺口
4. 休市时用户打开页面，发现最近 7 天历史不完整
5. 定时巡检发现某个 cache key 不完整

### 11.2 休市时补数的优先级反而更高
因为休市时用户看不到最新流，只能看历史。

所以休市时：
- 如果当前窗口缺历史
- 应主动发起补数
- 不要因为“现在没实时需求”就不补

### 11.3 补数状态要能落到 UI
用户必须能看见：

- 是否已发起 backfill request
- 当前 request 状态
- 哪些 segment 还没补齐
- 最近一次 ack 时间

建议顶部状态或图表提示使用：

- `补数：等待执行`
- `补数：ATAS 已接收`
- `补数：历史回填中`
- `补数：已完成`
- `补数：失败，待重试`

## 12. 前端显示与交互方案

### 12.1 主图不能黑屏无说明
如果当前没有 candles，可根据状态显示不同空态：

#### 空态 A：休市 + 无缓存
```text
当前市场休市，且当前窗口暂无可恢复历史。
你可以：
- 查看最近可用交易日
- 缩短时间范围
- 触发历史导入
```

#### 空态 B：补数中
```text
当前窗口历史尚未齐全，系统正在自动补数。
已恢复最近可用数据，缺失 240 根 bar。
```

#### 空态 C：构建失败
```text
图表快照构建失败。
请重试，或打开最近缓存查看。
```

### 12.2 状态条必须明确
顶部或图表上方建议固定显示：

- 数据模式：实时 / 历史 / 休市历史
- 完整性：完整 / 缺失 / 补数中
- 最近可用 bar 时间
- 最近同步时间
- 当前缓存来源：本地缓存 / 后端缓存 / 最新构建

### 12.3 图表上的缺段提示
对于中间缺段，不建议完全无提示。

建议：
- 允许 filler 维持时间轴连续
- 但鼠标 hover 或状态说明中能看到：
  - `该段为临时占位，等待真实历史回填`

### 12.4 “查看最近可用窗口”按钮
当用户选中的窗口无数据时，建议提供按钮：

- `跳到最近有数据的位置`
- `查看最近交易日`
- `切换到最近 1 天`

这是非常实用的降级操作。

## 13. 后端接口建议

建议新增或强化以下输出字段，不一定全新加接口，也可以并入现有 snapshot/live-tail 响应。

### 13.1 snapshot 响应必须带 missing-data 状态
例如：

```json
{
  "candles": [...],
  "integrity": {...},
  "display_status": {
    "market_state": "closed",
    "data_mode": "historical",
    "snapshot_state": "partial",
    "cache_state": "restored",
    "display_message": "当前市场休市，已显示最近可用历史；仍有缺口待补。"
  },
  "fallback_window": {
    "requested_start": "...",
    "requested_end": "...",
    "actual_display_start": "...",
    "actual_display_end": "..."
  }
}
```

### 13.2 建议新增最近可用窗口查询
例如：

- `GET /api/v1/workbench/latest-available-window?symbol=NQ&timeframe=1m`

返回：
- 最近存在数据的窗口范围
- 最近 bar 时间
- 推荐回退窗口

### 13.3 建议新增缺数据诊断接口
例如：

- `GET /api/v1/workbench/data-diagnostics?symbol=NQ&timeframe=1m&start=...&end=...`

返回：
- cache 命中情况
- snapshot 情况
- gap segments
- market state
- stream state
- fallback 建议

这对排查问题非常有价值。

## 14. 缓存专项要求

### 14.1 新开页面必须优先恢复最近状态
这点必须做成硬要求：

- 新页面不能总是从 0 开始
- 只要本地或后端存在最近成功快照，就应优先恢复

### 14.2 休市时缓存价值最大
因为此时：
- 没有新流
- 历史就是主内容
- 缓存几乎等于“首屏生命线”

所以休市时应优先：

1. 本地缓存秒开
2. 后端缓存兜底
3. 背景补数更新缓存

### 14.3 缓存失效不能直接导致空白
即使缓存过旧，也应该：
- 先展示旧缓存
- 明确标记：`缓存较旧，正在校准`
- 等后端新数据回来后再替换

而不是先空白再加载。

## 15. 可观测性与排障

缺数据问题如果没有观测，很难稳定解决。

### 15.1 后端应记录的关键日志
每次 snapshot build / live tail / backfill / fallback 恢复时，记录：

- symbol
- timeframe
- requested window
- actual display window
- market_state
- stream_state
- expected bars
- actual bars
- missing bars
- gap segments
- cache hit source
- backfill request id
- ack status

### 15.2 前端应能查看的数据诊断面板
建议在“更多”或“查看缓存”旁增加“数据诊断”：

展示：
- 当前页面状态
- 当前是否休市
- 当前是缓存恢复还是最新构建
- 最近 bar 时间
- 缺失 bar 数
- gap segments
- backfill 状态

这样用户和开发者都能快速判断问题，不用靠猜。

## 16. 验收标准

### 场景 1：开市中，实时正常
- 图表持续更新
- 状态显示 `实时 / 完整`
- 不误报缺数据

### 场景 2：开市中，尾部断流
- 状态切到 `实时（降级）`
- 自动发起补数
- 不黑屏

### 场景 3：休市，历史完整
- 页面默认显示最近可用历史
- 状态显示 `休市 / 历史完整`
- 不因为没有新流而空白

### 场景 4：休市，历史不完整
- 页面仍显示已有历史
- 状态显示 `休市 / 历史不完整 / 补数中`
- 用户知道当前不是“系统坏了”

### 场景 5：当前窗口无数据，但最近交易日有数据
- 页面给出“查看最近可用窗口”
- 支持一键跳转

### 场景 6：新开页面
- 优先恢复最近缓存
- 不从空白图开始
- 再静默追最新版本

### 场景 7：snapshot 构建失败
- 明确显示错误态
- 可回退最近缓存
- 可重试

## 18. 最终结论

你指出的“缺数据问题没补上，而且休市时更明显”，这个判断是对的，而且是**根问题**。

真正要解决的，不是简单地“再补一个 backfill 按钮”，而是把整个系统改造成：

- **能识别当前为什么缺数据**
- **能区分休市 / 断流 / 空窗口 / 构建失败 / 历史不完整**
- **能优先恢复最近可用历史，而不是黑屏等待**
- **能在休市时自动切到历史可用模式**
- **能把补数状态和可信度明确告诉用户**

一句话总结：

> 缺数据问题的正确解法，不只是“补数据”，而是“让系统在任何数据状态下都可恢复、可解释、可降级、可继续使用”。

这份文档就是后续专项改造的设计基线。
