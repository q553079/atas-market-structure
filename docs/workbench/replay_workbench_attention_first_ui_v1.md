# Replay Workbench Attention-First UI V1

状态：设计草案  
日期：2026-03-31  
范围：`Replay Workbench` 的事件流、AI 对话、文本与 K 线交互体验  
不在范围：识别引擎主路径、在线识别判定、自动交易

---

## 1. 这份文档要解决什么

当前 `Replay Workbench` 的 K 线主图能力已经基本可用，但事件能力、AI 交互、上下文透明度、文本与图表映射方式仍然明显偏向“实现堆叠”，而不是“人类使用习惯”。

用户真正的第一目标不是研究系统里有多少面板，而是：

1. 发一句话上来。
2. 很快看到 AI 对当前盘面的看法。
3. 必要时再追问它为什么这么看、用了哪些上下文、挂了哪些对象。

也就是说，界面应该按照注意力分配来设计，而不是按照内部模块数量来设计。

这份文档的目标是把这一方向正式定义清楚，并给出后续前端改造时必须遵守的交互原则、模块边界、展示规则和优化路线。

---

## 2. 核心结论

`Replay Workbench` 后续 UI 应该采用类似 `Codex` / `ChatGPT` 的注意力优先模型：

- 第一层只保留：`主图 + 输入 + 结构化回复`
- 第二层再暴露：`附近事件 + 上下文配方 + 挂载对象`
- 第三层才暴露：`Prompt Trace + 调参 + 调试治理`

这不是“把现有右侧面板做得更漂亮”，而是要把整套交互从“大杂烩”改成“渐进暴露”。

一句话描述目标产品：

> 用户先把 AI 当成一个会看盘、会对图、会讲重点的工作台，而不是一个堆满控制器、开关、面板、候选列表和隐藏逻辑的后台。

---

## 3. 当前问题总结

### 3.1 现在的界面是按模块摊开的，不是按任务收敛的

现有页面主结构同时长期占据：

- 主图区域
- 独立事件流列
- AI 侧边栏
- 事件判断 AI
- 事件整理
- Prompt 上下文条
- 挂载回复条
- 会话摘要
- Prompt Trace 模态

参考：

- `src/atas_market_structure/static/replay_workbench.html` 第 252-491 行
- `src/atas_market_structure/static/replay_workbench.css` 第 743-848 行

这会导致一个问题：用户第一眼根本不知道“现在最该看哪一块”。

### 3.2 分析入口重复，按钮语义冲突

当前 AI 区域既有：

- `analysisTypeSelect / analysisRangeSelect / analysisStyleSelect`
- `发送到当前会话 / 新会话发送`
- 快捷按钮 `最近20K / 最近20分钟 / 重点区域 / 实时挂单 / 插入选中K线 ...`

参考：

- `src/atas_market_structure/static/replay_workbench.html` 第 323-371 行

这里存在两个明显问题：

- 同一意图被多个入口重复表达
- 某些选项在特定周期上语义重复，例如 `1m` 图里 `最近20根K线` 和 `最近20分钟` 基本是一个东西

### 3.3 AI 回复展示仍然像“技术聊天记录”，不像“盘面结论卡”

当前主回复区域本质上还是聊天线程，虽然已经带了 `chip`、附件、计划卡、Prompt Trace 按钮，但核心仍然是“气泡消息流”。

参考：

- `src/atas_market_structure/static/replay_workbench_ai_threads.js` 第 736 行起
- `src/atas_market_structure/static/replay_workbench_ai_threads.js` 第 2437 行

目前的问题不是“能不能显示文字”，而是：

- 重点不够突出
- 结构不稳定
- 旧回复占屏
- 图文映射关系不够直接
- 用户很难一眼知道这条回复到底绑定了哪一段图

### 3.4 二级系统太多，用户脑内模型被拆碎

当前 AI 侧栏里除了主对话，还有：

- `事件判断 AI`
- `事件整理`
- `Prompt 上下文`
- `挂载回复`
- `会话摘要`

参考：

- `src/atas_market_structure/static/replay_workbench.html` 第 402-491 行

这会产生“多脑袋系统”问题：

- 主 AI 在说什么
- 事件 AI 在说什么
- 事件整理在提什么候选
- 事件流列又显示什么

用户很难建立统一心智模型。

### 3.5 当前列表渲染方式仍然偏粗糙，影响流畅度

现有代码已经能看出两个关键性能风险：

- AI 线程整体 `innerHTML = session.messages.map(...)`
- 事件流整体 `innerHTML = filteredCandidates.map(...)`

参考：

- `src/atas_market_structure/static/replay_workbench_ai_threads.js` 第 2420-2443 行
- `src/atas_market_structure/static/replay_workbench_event_panel.js` 第 708-715 行

这意味着只要消息、事件或状态变动，整个区域就可能整体重绘。  
对纯展示来说勉强能用，但对“盘中实时流 + 图文联动 + 鼠标交互 + 折叠动画”来说会很容易造成：

- 滚动不稳
- hover 状态抖动
- 选中状态闪动
- 收起展开不顺
- 阅读时内容突然换位

### 3.6 现有布局是“固定多列”，空间利用率天然吃亏

主布局现在是固定 grid：

```css
grid-template-columns: minmax(0, 1fr) minmax(120px, 150px) var(--resize-handle-width) minmax(520px, var(--sidebar-width));
```

参考：

- `src/atas_market_structure/static/replay_workbench.css` 第 743-749 行

这相当于默认长期保留：

- 主图大列
- 事件流窄列
- 拖拽列
- AI 侧栏大列

这套方式的缺点是：

- 不论用户当前要不要看事件流和深层 AI，都长期占空间
- 中间事件列太窄，信息密度难受
- 右侧 AI 栏太厚时，图就被明显挤压
- 屏幕宽度变化时，布局很容易显得局促和空白并存

## 4. 设计总原则

### 4.1 注意力优先

界面必须围绕用户的主任务，而不是内部系统结构。

默认优先级固定为：

1. `我问了什么`
2. `AI 现在怎么看`
3. `图上对应哪里`
4. `它为什么这么看`
5. `它到底用了什么 Prompt / 事件 / 记忆`

### 4.2 渐进暴露

默认界面只暴露高频能力。  
低频、专家、审计、调试能力必须下沉到第二层或第三层。

### 4.3 图文同构

AI 回复不是脱离图表存在的长文本。  
回复里的时间、对象、风险、结论都应该能够反向映射到图。

### 4.4 附近优先

界面前台只展示“当前窗口附近”和“仍影响当前窗口”的事件。  
老事件不是删除，而是退出前台展示。

### 4.5 显式上下文

AI 不能是黑盒。  
用户不一定天天想看 Prompt，但当用户想看时，必须能清楚知道本次回答用了什么上下文。

### 4.6 默认保守跨日

盘中临时上下文默认不过夜。  
只有明确固定的锚点、计划、长期设定，才允许跨日挂载。

### 4.7 AI 不进入识别主路径

必须继续遵守仓库约束：

- AI 只做辅助解释、整理、建议
- 不直接决定在线识别结果
- 识别主链路继续保持 deterministic

## 4.8 这份文档在实现前还必须补成“硬规格”

这份文档当前的主体仍然偏向方向说明。  
在进入真实前端改造前，以下四类内容必须先冻结为硬规则：

- 关键术语与状态定义
- Prompt 治理规则
- AI 谨慎输出协议
- 可量化验收指标

如果没有这四类硬规格，就会出现一个典型问题：

- 设计觉得“当前窗口”很清楚
- 前端觉得“附近事件”已经实现
- 后端觉得“跨日锚点”只是一个标签

最后同一个对象在不同模块里会被解释成不同意思。

因此，后续实现必须遵守一个顺序：

1. 先冻结术语、状态与边界。
2. 再改交互与视觉。
3. 再做性能和动画优化。

## 5. 目标交互模型：三层注意力结构

## 5.1 Layer 1：默认盘中视图

目标：最快完成一次“问一句 -> 得到结构化盘面答案”。

默认只保留：

- `Chart Workspace`
- `输入框`
- `AI Workspace`

用户在这一层应该能完成的事：

- 直接问一句当前盘面怎么看
- 快速看到 AI 的结构化结论
- 一眼知道结论对应的时间和图上对象
- 必要时点一下让图上对象高亮

不应该在这一层长期常驻的内容：

- Prompt block 列表
- 多组高级分析模板
- 事件判断 AI
- 事件整理大面板
- 长历史事件流
- 各种二级批量操作按钮

## 6. 主要模块定义

## 6.1 Chart Workspace

主图仍然是第一视觉中心。  
但是它不再承担“把所有说明全贴上去”的职责。

主图上只允许出现三类文字对象：

1. `结构关键对象`
   - 关键位
   - 风险区
   - 已确认区域

2. `当前回复相关说明`
   - 当前聚焦回复的短说明
   - 默认只显示当前聚焦回复，不显示整段历史说明

3. `附近事件标记`
   - 时间点
   - 小标签
   - 点击后反查到附近事件 dock 或回复卡

图上说明密度应提供固定策略：

- `仅图形`
- `图形 + 短标签`
- `仅当前回复说明`
- `隐藏全部说明`

默认推荐：`图形 + 短标签`

---

## 6.2 Input Composer

输入区应该像 Codex 一样干净。

默认输入区只保留：

- 一个输入框
- 一个主动作按钮：`发送`
- 一个上下文动作按钮：`分析当前窗口`

允许存在但应隐藏到二级菜单的动作：

- 附件
- 图表截图
- 模型切换
- 新建会话发送
- 调试入口

输入框的任务不是暴露所有控制，而是让用户顺畅说话。

推荐 placeholder：

- `现在盘面怎么看？`
- `这段上涨如果失效，失效点在哪里？`
- `结合当前窗口，给我简短结论和风险。`

---

## 6.3 AI Answer Card

AI 回复必须从“聊天气泡”进化为“结构化答复卡”。

### 固定结构

每张答复卡固定四段：

- `结论`
- `时间`
- `对象`
- `风险`

可选附加段：

- `计划`
- `证据`
- `下一步观察`

不是所有答复都必须给出强结论。  
因此每张答复卡头部还必须带一个显式状态：

- `assertion_level`

推荐四档：

- `观察性`
- `条件性`
- `高不确定`
- `上下文不足`

当 `assertion_level` 不是 `观察性` 时，`不确定性` 段不再是可选项，而是必须展示。

### 三态展示

#### Full

用于当前聚焦回复。

展示：

- 完整四段结构
- 对齐对象状态
- 回复时间窗口
- 图上对象数量
- 可选短说明

#### Compact

用于最近但未聚焦的回复。

展示：

- 一句结论
- 时间
- 风险短句

#### Skim

用于更旧的回复。

展示：

- 2 到 4 个字的小标题 chip
- 时间
- 是否上图

### 交互规则

- 点击卡片：主图高亮对应对象
- 点击图上对象：反向定位到该卡
- 视图切换后，旧卡不自动展开
- 新回复生成时，只允许当前卡流式展开，旧卡自动收紧

### 需要新增的元信息

每张回复卡都应有：

- `reply_window`
- `reply_window_anchor`
- `reply_session_date`
- `alignment_state`
- `assertion_level`
- `object_count`
- `stale_state`
- `context_version`

其中 `stale_state` 很重要：

- 如果图表窗口已经明显偏离这条回复的分析窗口，就应显示 `旧窗口`
- 用户仍可回看，但不会误以为它是“现在”的判断

---

## 6.4 Nearby Context Dock

`事件流` 不应该再是一整列长期独立系统，而应变成“附近上下文 dock”。

前台只显示三类事件：

1. `刚发生`
2. `仍在影响当前窗口`
3. `固定锚点`

默认保留范围建议：

- 当前可见窗口
- 再向前回推一日
- 只保留仍然和当前窗口逻辑相关的事件

前端展示策略：

- 离开当前窗口太远的历史事件，从前台清掉
- 后端事件流完整保留，不删除
- 拖回对应时间范围时，可重新出现

这和“数据库保留”并不冲突。  
前台清理只是注意力治理，不是数据删除。

---

## 6.5 Context Recipe

`Context Recipe` 是解决 AI 黑盒问题的关键模块。

它不需要默认完全展开，但必须始终可见一个摘要。

默认摘要显示：

- 当前窗口范围
- 已选 Prompt blocks 数量
- 是否包含记忆摘要
- 是否包含最近消息
- 是否挂载固定上下文
- 当前模型

用户展开后可以看到：

- 具体 block 顺序
- block 来源
- pinned / ephemeral 区别
- 本轮请求快照

这比把大量 prompt 芯片长期摊在主界面上更符合注意力习惯。

---

## 6.6 Mounted Reply / Object Strip

当前“已挂载回复 / 计划”这一层是有价值的，但不应该像一个独立工作面板。

应该改为轻量状态条：

- 当前挂载几条回复
- 当前主图显示哪些对象
- 是否存在跨日固定锚点

点击后才展开详情。

---

## 6.7 Prompt Trace

现有项目已经有 `Prompt Trace Modal`，这是正确方向，但它应该被明确定义为 `Layer 3`。

参考：

- `src/atas_market_structure/static/replay_workbench.html` 第 688-704 行

它不应该成为主界面常驻元素。  
它的职责是“解释这条回复为什么产生”，而不是承担主阅读功能。

---

## 6.8 Change Inspector

这里补充一个接近 `Codex` 右侧“切换差异面板”的模块，但它不能照搬代码 diff 逻辑。

在本项目里，这个模块更准确的名字应该是：

- `Change Inspector`
- 或 `变更面板`

它的职责不是比较两段长文本哪里改了几个字，而是回答下面这些真正有价值的问题：

- 为什么刚才 AI 的判断变了
- 这次改的是结论，还是只改了风险位
- 这次是不是多带入了新的上下文
- 这次新增引用了哪些附近事件
- 图上对象为什么和上一轮不一样

### 为什么不能做成普通文本 diff

`Codex` 的 diff 对象是代码，前后变化天然稳定。  
本项目的对象是：

- 盘面结论
- 事件引用
- 上图对象
- 上下文配方

如果直接做原始回复文本 diff，大概率会有几个问题：

- 噪音太大
- 同义改写会产生大量伪变化
- 用户看不出哪些变化真正影响决策
- 很容易又变成一个“技术感很强但实际不好用”的面板

所以这里必须明确：

> `Change Inspector` 只能做语义 diff，不能做原始长文本 diff。

### 应该比较什么

只比较结构化字段：

- `结论`
- `时间窗口`
- `对象`
- `风险`
- `失效条件`
- `已挂载图上对象`
- `引用事件`
- `Context Recipe 摘要`

推荐四个切换页签：

- `回复变化`
- `上下文变化`
- `事件变化`
- `对象变化`

### 不应该比较什么

- 不比较整段自然语言逐字差异
- 不比较跨很久以前的历史回复
- 不比较不同 symbol / timeframe 的回复
- 不把不影响交易理解的措辞改写当成变化

### 触发规则

这个面板只在“可比较”时出现：

- 与上一条同 symbol、同 timeframe、同 reply_window 的回复比较
- 或与上一条仍然有效的当前窗口回复比较

如果比较基础不成立，则不展示变更面板，只展示普通答复卡。

### 默认展示策略

这个面板不应常驻主界面。  
它应该和 Codex 类似，作为主回复卡上的一个二级切换入口：

- `答复`
- `变化`
- `上下文`
- `上图对象`

默认进入 `答复`。  
用户想追问“为什么变了”时，再进入 `变化`。

### 折叠行为

`Change Inspector` 必须是可折叠面板，而不是永久摊开的信息栏。

建议固定三态：

#### Collapsed

默认态。

展示：

- 一个很轻的入口标签
- 例如 `变化 3`
- 或 `有变更`

这一态只负责提示“这一轮和上一轮相比发生了结构性变化”，不抢主答复区空间。

#### Peek

轻展开态。

展示：

- 一列窄栏摘要
- 只显示最重要的 2 到 4 条变化
- 不展示完整解释

适合用户快速确认：

- 是结论变了
- 还是风险位变了
- 还是只多了一个事件引用

#### Expanded

完整展开态。

展示：

- `回复变化`
- `上下文变化`
- `事件变化`
- `对象变化`

每一类都允许查看完整语义变化记录。

### 默认规则

- 默认始终从 `Collapsed` 开始
- 用户主动点击 `变化` 后，进入 `Peek` 或 `Expanded`
- 关闭后回到 `Collapsed`
- 切回 `答复` 时，不应继续长期占据版面

### 与主答复卡的关系

必须明确优先级：

- `答复` 是主视图
- `变化` 是解释视图

这意味着：

- 新回复到来时，默认优先展示 `答复`
- 不自动把用户切到 `变化`
- `变化` 只在用户需要追因时才进入

### 窄屏与拥挤布局规则

在宽屏下，它可以像 Codex 那样作为右侧 inspector 拉开。  
在窄屏或空间紧张时，不应强行并排常驻。

推荐规则：

- 宽屏：右侧折叠 inspector
- 中等宽度：覆盖式抽屉
- 窄屏：底部抽屉或浮层

无论哪种形态，都必须保持：

- 默认折叠
- 一键关闭
- 不打断主图和主答复阅读

### 视觉草图

下面这张草图专门展示 `Change Inspector` 的三态：

- `Collapsed`
- `Peek`
- `Expanded`

同时也强调一个硬规则：

> 主答复永远优先，`Change Inspector` 只是按需展开的解释层。

![Change Inspector 草图](../../output/mockups/change_inspector_mockup_20260331.png)

### 一条好用的变更记录应该长什么样

不是这样：

- “将‘偏多延续’改为‘仍偏多但需确认’”
- “在第二段新增一句说明”

而应该是这样：

- `结论：震荡偏多 -> 延续偏多`
- `风险位：21531 -> 21536`
- `事件：新增引用 09:37 回踩确认`
- `对象：新增关键位标签 21542.25`
- `上下文：本轮额外挂载 1 个 pinned block`

### 优先级判断

这个模块是有价值的，但不是第一阶段。

只有在以下基础已经稳定之后，它才值得做：

1. `AI Answer Card` 已结构化
2. `Context Recipe` 已明确
3. `reply_window` 已稳定
4. 图上对象和回复绑定已稳定

否则它只会放大当前系统的混乱。

所以它的定位应是：

- `值得做`
- `但晚于结构化回复`
- `默认收起`
- `只做语义变化解释`

## 7. 事件与 AI 的关系应该如何重构

## 7.1 当前问题

现在系统里与“事件”相关的可视面至少有：

- 左中间 `事件流`
- 右侧 `事件判断 AI`
- 右侧 `事件整理`
- 主图 `chart-event-rail`
- 聊天消息上的事件 chip

这导致信息分散。

## 7.2 目标关系

后续应该把它收敛成一个统一链路：

1. 后端有完整正式 `EventCandidate`
2. 前台只投影“附近事件”
3. AI 回答时引用其中一部分事件
4. 回答产出对象后，再映射回图上
5. 用户查看时只看到“附近事件 + 当前回复引用了哪些事件”

也就是：

`后台完整事件流 -> 前台附近事件 -> AI 引用 -> 图上对象 -> 用户阅读`

而不是五个平行系统。

## 7.3 事件清理规则

### 前台规则

- 当前窗口内事件：保留
- 仍影响当前窗口的前序事件：保留
- 距离窗口过远且不再影响当前分析的事件：前台清掉

### 后台规则

- 永久保留完整事件
- 保留审计与回放能力

### 用户规则

- 默认只看附近
- 需要时才点 `查看全部历史`

## 8. AI 上下文边界与跨日策略

这是后续必须明确写死的规则。

## 8.1 默认不过夜

以下内容默认不过夜：

- 盘中临时问答
- 非 pinned prompt block
- 最近消息上下文
- 临时对象挂载
- 临时事件引用

## 8.2 允许跨日的只有明确锚点

允许跨日的内容应限制为：

- 手工固定关键位
- 手工固定风险结构
- 明确 pin 的上下文块
- 用户显式固定的计划卡

## 8.3 回复必须写明窗口

每条 AI 回复必须自带：

- 分析窗口
- 对应交易日
- 若为跨日引用，标记为 `跨日锚点`

这样用户就不会搞不清楚：

- 这是今天的判断
- 还是把昨天的临时上下文带进来了

## 8.4 关键术语与状态定义

下面这些词后续不能再只做产品语言，必须在实现时按同一语义落库或进 payload。

### `chart_visible_window`

定义：

- 当前主图真实可见的时间范围 `[visible_from, visible_to]`
- 以用户当前缩放和平移后的可见区间为准

规则：

- 这是系统里唯一合法的“当前窗口”来源
- 不能由其他面板自行推测

### `reply_window`

定义：

- 本条 AI 回复实际分析所使用的绝对时间区间 `[window_start, window_end]`

规则：

- 发送时冻结
- 生成后不因用户继续拖图而回写

### `reply_window_anchor`

定义：

- `symbol + timeframe + window_start + window_end + session_date`

作用：

- 作为回复、图上对象、Change Inspector 的共同比较键

### `nearby_event`

定义：

- 事件时间落在 `chart_visible_window` 内
- 或虽不在可见窗口内，但满足 `influencing_event`

### `influencing_event`

定义：

- 位于当前窗口之前
- 且仍然满足以下至少一条：
  - 被当前 `active_reply` 引用
  - 绑定了当前仍可见的图上对象
  - 是用户显式固定的锚点

### `fixed_anchor`

定义：

- 用户显式固定、允许跨窗口或跨日保留的关键位、风险结构、上下文块或计划卡

规则：

- 没有显式固定，就不能被视为 `fixed_anchor`

### `pinned_context`

定义：

- 用户主动固定并选择继续参与后续回答的上下文块

### `ephemeral_context`

定义：

- 系统自动生成或仅对本轮有效的上下文块

规则：

- 默认不过夜
- 默认不进入跨日比较

### `active_reply`

定义：

- 当前用户正在阅读、聚焦、或与图表联动的那一条回复

规则：

- 同一 `symbol + timeframe` 工作区同时只能有一条 `active_reply`

### `stale_state`

定义：

- `当前窗口`
  - 同 `symbol / timeframe`
  - 且 `reply_window` 与 `chart_visible_window` 的重叠比例不低于 50%
- `旧窗口`
  - 同 `symbol / timeframe`
  - 但重叠比例低于 50% 或已无重叠
- `跨日引用`
  - 本条回复使用了上一交易日的 `fixed_anchor`
- `需要刷新`
  - 当前 `symbol / timeframe` 已变化
  - 或存在更新的同范围回复，而本条回复与当前窗口已无重叠

## 8.5 Prompt 治理规则

要解决 AI 黑盒问题，不能只让用户“看见 prompt”，还必须明确哪些 prompt 可控、哪些不可控。

### Prompt block 分类

- `system_policy`
  - 系统规则，用户只读
- `analysis_template`
  - 分析模板，用户只读或只可切换
- `window_snapshot`
  - 当前窗口快照，系统生成，只读
- `nearby_event_summary`
  - 附近事件摘要，系统生成，可查看来源
- `memory_summary`
  - 记忆摘要，系统生成，可开关
- `recent_messages`
  - 最近消息摘要，系统生成，可开关
- `user_note`
  - 用户手写说明，可编辑
- `pinned_context`
  - 用户固定上下文，可编辑、可固定、可取消固定

### 可编辑性矩阵

- `system_policy`
  - 不可编辑
- `analysis_template`
  - 不直接编辑，只切换模板
- `window_snapshot`
  - 不可编辑
- `nearby_event_summary`
  - 不可编辑，但必须可追溯来源
- `memory_summary`
  - 只能开关
- `recent_messages`
  - 只能开关
- `user_note`
  - 可编辑
- `pinned_context`
  - 可编辑、可版本化

### 版本规则

每个可编辑块必须带：

- `block_id`
- `block_version`
- `author`
- `updated_at`
- `scope`

每次发送请求时，必须记录：

- 选中的 block id
- 对应 block version
- 是否为 pinned

这样 `Prompt Trace` 才能回答：

- 这条回复到底看到了哪个版本
- 后来用户改过没有

### 回滚规则

对于 `user_note` 和 `pinned_context`，至少要支持：

- 查看上一版本
- 回滚到上一版本
- 清楚标记“当前回复使用的不是最新版本”

### 跨日范围规则

`pinned_context` 应至少支持三种 scope：

- `session`
- `trading_day`
- `manual_global`

默认推荐：

- 非显式固定内容：不过夜
- `pinned_context`：按用户选定 scope 生效

## 8.6 AI 谨慎输出协议

本项目是交易分析工作台，不是一般聊天产品。  
因此 AI 的表达方式必须比通用聊天更保守。

### 每条回复必须携带 `assertion_level`

四档定义：

- `观察性`
  - 只描述当前结构和图上对象
- `条件性`
  - 给出判断，但必须绑定明确失效条件
- `高不确定`
  - 允许总结，但必须突出不确定来源
- `上下文不足`
  - 禁止给方向性结论，只能说明缺什么

### 必须降级的场景

出现以下情况时，不能给“强结论”，必须降级为 `条件性`、`高不确定` 或 `上下文不足`：

- 没有稳定的 `reply_window`
- `alignment_state` 主要为 `模糊` 或 `待确认`
- 存在跨日引用但没有明确标记
- 当前窗口信息不足以支撑因果判断
- 用户问题超出了当前挂载上下文

### 必须拒绝硬下结论的场景

以下情况下，只能返回 `上下文不足`：

- 当前没有图表窗口快照
- `symbol / timeframe` 与当前工作区不一致
- 关键上下文块缺失且无法自动补齐

### 强结论的最低要求

只有当回复满足以下条件，才允许给出明确方向或交易倾向：

- 有明确 `reply_window`
- 有可见图上对象或事件支撑
- 有失效条件
- `assertion_level` 不是 `高不确定` 或 `上下文不足`

### UI 展示要求

当 `assertion_level` 为：

- `条件性`
  - 必须显式展示 `失效条件`
- `高不确定`
  - 必须显式展示 `不确定性`
- `上下文不足`
  - 禁止展示计划卡、禁止展示方向性标题

## 10. 视觉与动效方向

## 10.1 整体风格

整体风格应继续朝你已经认可的方向推进：

- 简约但不空
- 重点明确
- 类似 Codex / ChatGPT 的工作型界面
- 模块干净，不像仪表盘拼装页

## 10.2 文字层级

AI 回答区需要明显层级：

- 标题：高对比、偏粗
- 结论：最醒目
- 风险：暖色强调
- 时间 / 对象：中强度
- 次要说明：低对比

不要再把所有内容都做成同一层级的 12px 段落。

## 10.3 动效原则

动效应该服务阅读，而不是装饰。

推荐：

- 新答复卡进入时做轻微淡入和位移
- Full -> Compact 用高度过渡
- Compact -> Skim 用收缩加 chip 化
- 图上对象高亮时，使用低频脉冲或边框闪现

避免：

- 整栏抖动
- 大面积布局位移
- 因整列表重绘造成的 hover 丢失

## 10.4 背景与色彩

维持深色主基调没问题，但要更注意分层：

- Shell
- Panel
- Card
- Active Card
- Chart Annotation

这些层次必须通过背景、边框、阴影、文字权重共同建立，不要只靠一种蓝色。

## 11. `Pretext` 能帮什么，不能帮什么

对 `chenglou/pretext` 的结论需要明确写下来，避免后续把它当万能药。

## 11.1 它适合做的事

- 文本精确测量
- 多段内容 shrink-wrap
- 收起展开时的低抖动高度计算
- 结构化答复卡的平滑布局
- 图上标签、注释、说明块的尺寸稳定

## 11.2 它不解决的问题

- 不会自动重构整个页面信息架构
- 不会修复事件流和 AI 面板的模块混乱
- 不会替代 DOM 增量更新策略
- 不会自动解决图文映射或上下文治理

## 11.3 适合引入的位置

如果后续要用它，应优先考虑：

- `AI Answer Card`
- `Context Recipe` 展开区
- 图上说明标签
- `Compact / Skim` 过渡时的内容测量

不建议一开始就试图让 `Pretext` 接管整个工作台布局。

## 12. 还能继续优化的地方

这一节是基于当前草图和现有代码，进一步往前多想一步。

## 12.1 回复应该有“新鲜度”状态

当市场继续走、或者用户已经把窗口拖到了别处，旧回复不应该还以“当前结论”的姿态存在。

建议增加：

- `当前窗口`
- `旧窗口`
- `跨日引用`
- `需要刷新`

这样可以减少误读。

## 12.2 线程应该弱化，任务应该强化

当前有明显“会话管理界面”的味道。  
但对用户来说，他更像是在做一件事：看当前盘面。

建议：

- 会话 tabs 降级成次要导航
- 当前工作对象改成“当前分析任务”
- 会话仍然存在，但不要成为表层主结构

这和 Codex 的思路一致：线程是组织单元，但不是用户每秒都要盯着的主角。

## 12.3 `事件判断 AI` 很可能不该继续作为独立用户面板存在

它更像内部协同能力，而不是用户主工作区。

更合理的方式是：

- 要么变成主 AI 的一个内部能力
- 要么进入第三层专家工具页

否则用户会一直困惑：

- 主 AI 跟事件 AI 哪个才算最终答案

## 12.4 `事件整理` 也不该继续像第二条主线程那样存在

它更适合作为：

- 附近事件 dock 的一个详情展开态
- 或第三层里的治理工具页

而不是常驻在主阅读区下面。

## 12.5 旧回复不应该永久占据滚动高度

如果盘中使用频率高，答复会越来越多。  
不管有没有虚拟列表，首先都应该先做“认知压缩”：

- 旧卡自动进入 `Skim`
- 同主题连续答复可折叠成组
- 明显过期的卡可以按窗口分组

这能比单纯上虚拟化更直接改善可用性。

## 12.6 需要一套统一的“对象对齐状态”

AI 说了很多东西，但用户最关心的是：

- 它到底对到了哪
- 有没有飘

建议统一定义四态：

- `已对齐`
- `模糊`
- `越界`
- `待确认`

并让每条回复、每个图上对象、每个事件引用都共享这一状态语义。

## 12.7 需要一套统一的“图上说明密度”策略

图表上最怕两个极端：

- 什么都没有，不知道 AI 在说哪
- 什么都有，屏幕被文字淹没

所以必须把图上说明变成统一策略，而不是各模块自己决定要不要画字。

## 12.8 上下文不应只可见，还应可解释

仅仅告诉用户“本轮用了 3 个 block”还不够。  
用户还会问：

- 为什么用了这 3 个
- 哪个是系统生成的
- 哪个是我固定的
- 哪个是因为最近消息自动带上的

所以 `Context Recipe` 后续最好增加来源分类：

- `用户固定`
- `系统自动挂载`
- `临时窗口生成`
- `跨日锚点`

## 12.9 回复和图表窗口应该建立稳定绑定

后续需要引入一个简单但关键的概念：

`reply_window_anchor`

也就是：

- 这条回复对应哪个时间窗口
- 这条回复是对哪个 symbol / timeframe 生效

这样做能避免窗口拖动后所有东西都显得悬空。

## 12.10 首屏不应该再出现多条并列工作流

当前首屏至少有：

- 图表工作流
- 事件流工作流
- AI 工作流
- 事件整理工作流

后续应收敛为：

- `看图`
- `问 AI`
- `看附近上下文`

剩下都是二级或三级。

## 13. 与当前代码结构的对应改造方向

这一节不改代码，只定义未来实现时的大方向。

## 13.1 `replay_workbench.html`

当前问题：

- 顶层结构上把多个系统长期并列摆开
- AI 侧栏内容层里又继续堆叠多块子系统

真正落地时建议分阶段做，不要一次性大改。

## Phase 0：先冻结术语、状态和契约

- 冻结 `assertion_level` 与降级规则
- 冻结 `Change Inspector` 的比较键和出现条件

输出物：

- 事件、回复、上下文的 additive metadata 契约
- 前后端共享的状态语义表

## Phase 1：先改首屏信息架构

目标：

- 首屏只保留主图、输入、答复卡、附近事件

边界：

- 只改信息层级和可见性
- 不先引入复杂动画

## Phase 2：答复卡化与谨慎输出

目标：

- assistant bubble -> structured answer card
- 加入 `Full / Compact / Skim`
- 加入 `assertion_level`
- 把 `失效条件 / 不确定性 / 上下文不足` 规则落实到 UI

## Phase 3：附近事件 dock 与窗口绑定

目标：

- 把事件流列改成窗口附近上下文区
- 统一图上对象和事件引用

## Phase 4：Context Recipe、Prompt 治理与 Trace

## 15. 这套方案明确不建议做什么

- 不建议继续往首屏增加更多长期常驻面板
- 不建议把更多分析模板直接暴露成按钮
- 不建议让多组 AI 并列争夺主结论
- 不建议默认跨日带入临时上下文
- 不建议把所有说明都直接贴在图上
- 不建议用 `Pretext` 代替信息架构重构

## 16. 最终设计判断标准

未来改造是否成功，不看“加了多少功能”，而看下面这些可验收条件是否成立。

## 16.1 主路径效率

- 用户提交消息后，`pending` 状态或结构化答复骨架应立即出现，不允许界面空白等待
- 默认首屏主动作不应超过 `6` 个
- 用户不展开二级能力时，首屏长期常驻模块不应超过 `4` 个

## 16.2 视图稳定性

- 同一 `symbol / timeframe` 下的普通刷新、AI 输出、上下文展开，不得重置主图 `chart_visible_window`
- 只有以下场景允许重置到新默认位置：
  - 用户切换 `symbol`
  - 用户切换 `timeframe`
  - 用户显式点击 `重置视图`
- 单条消息更新、单条事件更新时，不应触发整列表全量重建
- 增量更新后，用户当前 scroll、hover、selected、focused 状态必须保留

## 16.3 噪音控制

- 默认前台 `Nearby Context` 可见事件数不应超过 `8`
- 默认完全展开的历史答复不应超过 `2`
- 更旧答复必须自动进入 `Compact` 或 `Skim`
- `Change Inspector` 在冷启动和普通阅读场景下应 `100% 默认折叠`

## 16.4 上下文透明度

- 每条答复必须展示：
  - `symbol`
  - `timeframe`
  - `reply_window`
  - `assertion_level`
- 一旦存在跨日引用，必须有可见的 `跨日锚点` 标记
- `Prompt Trace` 必须能追溯到具体 block version

## 16.5 谨慎输出安全性

- 只要出现 `高不确定` 或 `上下文不足`，就不能把答复渲染成强结论样式
- 只要给出方向性结论，就必须同时给出失效条件
- `上下文不足` 状态下，禁止渲染计划卡

## 16.6 定性检查

除了上面的硬指标，最终还应满足以下定性判断：

1. 用户打开页面后，是否一眼知道先做什么。
2. 用户发一句话后，是否很快抓到 AI 的结构化结论。
3. 用户能否一眼看出这条回复对应主图哪里。
4. 用户不看细节时，界面是否依然干净，不被旧文本和老事件吞没。
5. 用户想深挖时，是否能清楚看到 Prompt、上下文、事件引用和对齐状态。
6. 窗口移动、刷新、盘中变化时，界面是否仍然稳定，不乱跳、不重置、不抢注意力。

如果硬指标和定性检查都成立，这套 UI 才算真的从“能用”进入“好用”。
