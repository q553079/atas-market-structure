# Replay Workbench 右侧 AI 聊天工作区设计文档（消息流核心版）

> 目标：把 Replay Workbench 的右侧区域从“带一些按钮的聊天框”，升级为一个真正可用于交易分析、复盘、计划管理、图表联动的 **AI 会话工作台**。
>
> 本文档聚焦：
> - 右侧 AI 聊天区的产品定位
> - 多轮对话与多聊天页能力
> - 不同品种 / 不同聊天窗 / 不同请求的隔离机制
> - Prompt 可见化与手动装配机制
> - AI 回复与右侧 K 线标记联动机制
> - 流式输出机制
> - 前后端对象设计
> - 组件拆分与开发优先级

---

# 1. 产品定位

右侧 AI 区不应是附属说明栏，也不应是简单问答框。

它应是：

**一个以消息流为核心、具备多会话、多品种隔离、Prompt 装配、流式输出、图表挂载能力的 AI 即时协作工作台。**

核心原则：

1. **消息流是右侧主战场**
   - AI 回复内容是核心生产力
   - 不是工具条，不是配置面板
   - 回复区必须占右侧最大面积

2. **会话像微信聊天一样自然**
   - 不同聊天像和不同人聊天
   - 可切换、可搜索、可置顶、可新建
   - 每个聊天独立保留自己的历史、草稿、挂载状态

3. **不同品种必须硬隔离**
   - NQ 的对话只属于 NQ
   - ES 的对话只属于 ES
   - 切换品种时必须进入一个新的空白聊天窗口
   - 不能自动继承别的品种上下文

4. **新聊天默认空白，不自动塞上下文**
   - 不自动发送当前行情
   - 不自动带最近20根K线
   - 不自动带截图
   - 不自动带历史摘要
   - 所有上下文由用户主动选择发送

5. **Prompt 必须可见、可审阅、可控制**
   - 用户能看到本次将发给 AI 的内容组成
   - 前端显示摘要版
   - 后端发送完整版

6. **AI 回复不只是文本，还要可投影到图表**
   - 每条 AI 回复都可以派生图表对象
   - 用户可选择哪条回复上图
   - 用户可选择一条回复中的哪些对象上图

7. **AI 输出必须流式传输**
   - 实时出字
   - 像成熟聊天产品一样连续展示
   - 支持中断、完成、失败等状态

# 2. 页面整体结构（右侧 AI 区）

## 2.1 总体布局原则

右侧 AI 区采用：

**轻头部 + 压缩上下文条 + 大消息流主区 + Prompt 发送条 + 固定输入区**

其中：
- 消息流必须占最大比例
- 其他一切都应让位给消息流

## 2.2 推荐高度占比

默认状态下建议：

- 顶部头部：`6% ~ 8%`
- 压缩上下文条：`5% ~ 7%`
- **消息流主区：`60% ~ 75%`**
- Prompt 发送条：`4% ~ 6%`
- 输入区：`10% ~ 15%`

说明：
- 消息流必须始终是最大区域
- Prompt 装配器默认不大展开
- 分析模板不得长期占据大面积

# 4. 多轮对话与多聊天页设计

## 4.1 设计目标

AI 工作区必须支持：

1. **多轮对话**
   - 同一会话中连续问答
   - 保留上下文，但不是强制全部继承
   - 每轮消息清楚展示状态

2. **多聊天页 / 多会话**
   - 像微信一样切换不同聊天
   - 每个聊天独立存在
   - 可搜索、置顶、删除、归档、复制分支

3. **多品种聊天共存**
   - 会话列表里可以保留 NQ、ES、CL 等不同品种会话
   - 但当前活跃会话必须与当前品种严格对应

---

## 4.2 会话列表形态

建议采用：

**左侧窄会话栏 + 中间消息流主栏**

如果当前阶段不想重做成双栏，也至少要把现有横向 tab 升级为可滚动、可搜索、可分组的会话列表。

### 推荐会话项结构

```text
┌────────────────────┐
│ ★ 会话03           │
│ NQ · 1m · 最近7天  │
│ 等回踩确认，不追高 │
│ 草稿中…        2   │
└────────────────────┘
```

字段建议：
- 标题
- 品种
- 周期
- 最后消息摘要
- 未读数
- 草稿状态
- 是否置顶
- 是否有 active plan

---

## 4.3 会话基本操作

每个会话支持：
- 切换
- 搜索
- 新建
- 重命名
- 固定/取消固定
- 删除
- 归档
- 复制分支

### 复制分支的作用
适合交易分析场景：
- 一个会话看回踩剧本
- 一个会话看突破剧本
- 一个会话给另一个模型重判

# 5. 不同品种、不同聊天窗、不同请求的隔离规则

这是本设计的关键规则。

## 5.1 品种隔离

不同品种必须硬隔离。

### 规则
- NQ 会话只能属于 NQ
- ES 会话只能属于 ES
- CL 会话只能属于 CL
- 不允许自动把 NQ 的历史、摘要、计划、图表对象带到 ES

### 切换品种时行为
例如从 `NQ` 切到 `ES`：

必须执行：
1. 保留 NQ 历史会话在列表中
2. 保留 NQ 消息记录
3. 但当前主区域必须打开一个 **新的 ES 空白聊天窗口**
4. 不自动带 ES 当前行情
5. 不自动带 NQ 的历史上下文
6. 输入框为空
7. Prompt 装配器为空

即：

**切换品种 = 进入该品种的新空白聊天，而不是继承现有聊天。**

---

## 5.2 聊天窗隔离

即便同为 NQ，不同聊天窗之间也必须严格隔离。

例如：
- 会话A：回踩做多
- 会话B：突破追多
- 会话C：最近20K微结构分析

它们之间不得自动共享：
- 历史消息
- 草稿
- Prompt 选择
- AI 摘要 memory
- 已挂载回复
- 当前主计划卡

---

## 5.3 单次请求隔离

同一个会话中，每次发送也必须明确区分：
- 本次到底发了哪些内容
- 哪些只是候选上下文
- 哪些没有发送

不能自动把“上次用过的上下文”默认塞进本次发送，除非用户明确固定。

---

# 6. Prompt 可见化与手动装配机制

## 6.1 目标

Prompt 不是黑盒，不应在后台偷偷拼装。

应做到：
- 用户能看到本次发送内容
- 用户能选择要不要发送某个上下文块
- 用户能看到摘要版预览
- 后端真正发送完整版数据

---

## 6.2 Prompt 装配器结构

Prompt 装配器是输入区上方的一个可折叠区块。

### 默认状态
压缩成一行：

```text
本次发送：用户输入 + 最近20K + 事件摘要 [展开]
```

### 展开状态

```text
┌──────────────────────────────────────────────┐
│ 本次发送内容（Prompt 预览）                  │
│ 已选择 4 项                                   │
│ [查看原始发送内容] [清空]                     │
├──────────────────────────────────────────────┤
│ ☑ 用户问题                                    │
│ ☑ 最近20根K（预览简写）                      │
│ ☑ 最近3小时行情（预览简写）                  │
│ ☑ 事件数据（简写预览）                        │
│ ☐ 当前图表截图                                │
│ ☐ 当前选中K线                                 │
│ ☐ 手工区域A                                   │
│ ☐ 当前活动计划                                │
└──────────────────────────────────────────────┘
```

---

## 6.3 候选上下文池

系统可以生成候选上下文，但默认不发送。

候选上下文示例：
- 最近20根K线
- 最近3小时行情
- 当前窗口
- 当前选中K线
- 手工区域
- 当前图表截图
- 重点区域
- 事件摘要
- 当前活动计划
- 当前会话摘要

### 默认规则
#### 新会话默认
- 只发用户输入文本
- 其他全部不选

#### 快捷分析按钮触发时
例如点“最近20K”：
- 自动勾选“最近20根K” block
- 但用户依然能看到这个 block
- 仍可手动移除或补充其他 block

---

## 6.4 前端显示简写，后端发完整版

例如“事件数据”：

### 用户可见预览
```text
事件数据（简写预览）
- 09:41 吸收
- 09:43 大单撤退
- 09:46 补单
共 12 条事件，已高亮重点 3 条
```

### 后端实际发送
依然发送完整结构化数据：
- 完整时间戳
- 类型
- price
- size
- side
- confidence
- raw metadata

# 7. AI 回复消息流设计

## 7.1 核心原则

消息流是右侧核心区域，必须：
- 宽
- 稳
- 可连续多轮阅读
- 支持流式输出
- 支持消息级图表操作

## 7.2 消息流布局要求

1. AI 消息宽度应比普通 IM 更宽
2. 长文不应是窄气泡
3. AI 回复宜采用卡片式结构，而不是单纯气泡
4. 消息流必须独立滚动
5. 输入区固定在底部
6. 允许自动吸底，但不能打断用户阅读

---

## 7.3 AI 消息卡结构

```text
┌──────────────────────────────────────────┐
│ AI #12  回踩做多逻辑                     │
│ 这里不追多，优先等 24852-24848 回踩确认  │
│                                          │
│ 关键对象 4 个                            │
│ ☑ 支撑区 24840-24848                     │
│ ☑ 无交易区 24856-24858                   │
│ ☑ 做多计划 A                             │
│ ☐ 路径推演箭头                           │
│                                          │
│ [上图] [仅本条] [叠加] [替换] [查看图表] │
│ [提取计划卡] [查看对象详情]              │
└──────────────────────────────────────────┘
```

字段建议：
- 消息编号
- 模型名（可选）
- 消息状态
- 正文
- 对象清单
- 图表操作按钮
- 结构化操作按钮

---

## 7.4 用户消息卡

用户消息保持简洁，但要可引用：
- 内容
- 发送时间
- 引用来源（如果是引用某条 AI 回复或某个图表对象）

---

# 8. AI 回复与 K 线标记联动机制

## 8.1 目标

每条 AI 回复不仅是文本，也是一个可投影到图表的分析单元。

### 一条 AI 回复应拆成两层
1. **文本层**
   - 自然语言结论
   - 推理说明
   - 风险说明

2. **对象层**
   - 支撑区
   - 阻力区
   - 无交易区
   - 开仓计划
   - 止损线
   - 止盈线
   - 路径箭头
   - 观察区

---

## 8.2 回复级上图控制

用户必须可以：
- 只显示某条回复的对象
- 叠加显示多条回复对象
- 替换当前显示对象
- 只显示该回复中的部分对象

### 每条回复建议支持操作
- `[上图]`
- `[仅本条]`
- `[叠加]`
- `[替换]`
- `[查看图表]`
- `[提取计划卡]`

---

## 8.3 回复对象勾选清单

用户可在消息内部细粒度控制：
- 只显示支撑阻力
- 不显示路径箭头
- 只显示计划，不显示无交易区

这是降低图表混乱的关键。

---

## 8.4 已挂载回复条（Mounted Reply Strip）

建议在消息流下方、输入区上方增加一个轻量区域：

```text
已挂载到图表的回复
[#12 回踩逻辑 👁] [#13 突破逻辑 👁] [清空] [仅保留当前]
```

作用：
- 告诉用户当前图表上到底显示了哪几条 AI 回复的对象
- 比标记管理器更轻
- 比直接在消息里找更直观

---

## 8.5 从聊天到图表

点击消息中的 `[上图]` / `[仅本条]` 后：
- 图表跳到相关时间区域
- 高亮该消息派生对象
- 图表只显示对应对象（若是仅本条）

---

## 8.6 从图表到聊天

点击图上的 AI 标记，弹出卡片：

```text
支撑区 24840-24848
来源：会话03 / AI回复#12
状态：triggered
[查看原回复] [只显示同源对象] [隐藏]
```

点击 `查看原回复`：
- 聊天滚动定位到原消息
- 原消息高亮数秒
- 自动展开该消息的对象列表

---

# 9. AI 流式输出机制

## 9.1 目标

AI 回复必须实时输出，不应等待整段生成完再一次性显示。

表现要接近成熟聊天产品：
- 先插入占位消息
- 持续追加 token
- 消息末尾显示光标/流式标记
- 支持完成、中断、失败

---

## 9.2 消息生命周期状态

### 用户消息状态
- `draft`
- `queued`
- `sending`
- `sent`
- `failed`

### AI 消息状态
- `pending`
- `streaming`
- `completed`
- `interrupted`
- `failed`
- `regenerated`

---

## 9.3 流式展示规则

### 当用户发送后
立刻插入一条 AI 占位消息：
- 头像
- 模型名
- “思考中…” / “正在分析…”

### 流式进行中
- 后端持续返回 token
- 前端把 token 追加到最后一条 AI 消息
- 如果用户在底部附近，则自动滚动到底部
- 如果用户上滑查看历史，则暂停自动吸底

### 流结束后
统一补充：
- 结构化对象
- 计划卡
- 摘要更新
- 可挂载回复对象

说明：
- 不建议流中频繁插入完整对象卡，容易抖动
- 推荐“先流正文，流结束后再落结构化对象”

---

# 10. 输入区设计

## 10.1 输入区永远固定
输入区不能因为折叠而消失。

必须包含：
- 文本输入框
- 附件入口
- 截图入口
- 插入选中 K 线
- 插入手工区域
- 发送按钮

## 10.2 输入框能力
应支持：
- 自由文本
- `@对象`：引用图表对象
- `/技能`：调用分析模板
- 引用上一条消息/结论

## 10.3 每会话独立草稿
每个会话单独保存：
- 草稿文本
- 草稿附件
- 已选择 Prompt block
- 上次展开状态

---

# 11. 会话记忆与上下文保留策略

## 11.1 默认策略：不自动带历史
新会话默认：
- 不带最近历史
- 不带 memory summary
- 不带当前行情
- 不带计划卡

## 11.2 需要延续时，由用户显式选择
例如：
- 保留最近 3 轮
- 保留会话摘要
- 保留当前主计划
- 保留挂载对象

## 11.3 切换 AI 时也遵循可控原则
切换模型时可选：
- 仅当前问题
- 当前问题 + 会话摘要
- 当前问题 + 最近 3 轮 + 会话摘要

默认不要偷偷全量重发。

# 13. 流式传输协议建议

## 13.1 推荐方式
优先建议：

**SSE（Server-Sent Events）**

优点：
- 实现简单
- 适合单向 AI 文本流

# 18. 给实现者的最终说明

这个系统的关键，不是多几个按钮，而是建立三套清晰边界：

## 第一套边界：消息流优先
- AI 回复内容必须是右侧最大区域
- 所有控制区都应压缩、折叠、让位给消息流

## 第二套边界：上下文可控
- Prompt 必须可见
- 用户必须知道自己发了什么
- 不允许系统偷偷自动塞过多上下文

## 第三套边界：隔离清晰
- 不同品种隔离
- 不同聊天窗隔离
- 不同请求隔离
- 回复对象与图表显示范围清晰可控

如果严格按这份文档实现，最终得到的将不是“一个带聊天框的图表页”，而是：

**一个以 AI 消息流为核心、支持多会话、多品种隔离、Prompt 可控、回复可上图、流式输出、可回跳图表的专业交易 AI 工作台。**
