# Replay Workbench UI / AI 交互状态表

> 目标：把 `replay_workbench_ui_ai_design.md` 与 `replay_workbench_ui_ai_schema_draft.md` 的设计，进一步落实成前端可直接实现的交互状态表。
>
> 适用对象：
> - 前端工程师
> - 交互实现型 AI
> - 测试工程师
> - 产品验收

---

## 1. 使用说明

本文件关注的是：

1. 用户执行了什么动作
2. 前端应该更新哪些状态
3. 后端可能需要触发什么请求
4. UI 应该呈现什么反馈
5. 哪些状态需要持久化

建议把每一条都视为：
- 一个交互场景
- 一个实现清单
- 一个测试用例模板

---

## 2. 全局状态域

本页面主要状态域分为：

- `workbench_context`
- `session_state`
- `message_state`
- `draft_state`
- `annotation_state`
- `filter_state`
- `layout_state`
- `model_state`
- `drawer_state`

## 3. 页面初始化与恢复

### 场景 3.1：页面首次加载

| 项目 | 内容 |
|---|---|
| 用户动作 | 打开 Replay Workbench 页面 |
| 前端动作 | 请求工作台快照 |
| 后端请求 | `GET /api/replay-workbench/snapshot` |
| 状态更新 | 初始化 `ReplayWorkbenchState` |
| UI反馈 | 显示上次品种、周期、窗口、会话、布局 |
| 持久化 | 不变 |

### 场景 3.2：恢复上次布局

| 项目 | 内容 |
|---|---|
| 用户动作 | 点击“恢复上次布局” |
| 前端动作 | 读取本地存储布局状态 |
| 状态更新 | 更新 `ui_state.layout` |
| UI反馈 | 左右栏宽度、底部抽屉、图层状态恢复 |
| 持久化 | 覆盖当前布局缓存 |

| 用户动作 | 点击“上下文/手工区域/焦点区域...” |
| 前端动作 | 更新 active drawer tab |
| 状态更新 | 抽屉 tab 改变 |
| UI反馈 | 内容区域切换 |
| 持久化 | 保存最后展开 tab |

## 14. 错误与边界状态

### 场景 14.1：发送消息失败

| 项目 | 内容 |
|---|---|
| 触发 | chat/send 返回错误 |
| 前端动作 | 将 assistant loading 变为错误态 |
| 状态更新 | message.error = true（前端内部态） |
| UI反馈 | 显示“发送失败，可重试” |
| 持久化 | 保留草稿，不清空 |

### 场景 14.2：切换 AI 失败

| 项目 | 内容 |
|---|---|
| 触发 | switch-model 返回错误 |
| 前端动作 | 回滚 active_model 展示 |
| 状态更新 | 不更新会话 active_model |
| UI反馈 | 显示“切换失败” |
| 持久化 | 保持原模型 |

### 场景 14.3：标记对象状态更新失败

| 项目 | 内容 |
|---|---|
| 触发 | annotations/update 或 evaluate 失败 |
| 前端动作 | 保留旧状态，提示用户刷新 |
| 状态更新 | 回滚 optimistic update |
| UI反馈 | toast 提示 |
| 持久化 | 以旧状态为准 |

### 场景 14.4：会话摘要生成失败

| 项目 | 内容 |
|---|---|
| 触发 | SessionMemory 生成失败 |
| 前端动作 | 允许切换 AI，但提示采用简化模式 |
| 状态更新 | handoff 降级 |
| UI反馈 | 提示“摘要生成失败，已退回最近消息模式” |
| 持久化 | 记录失败日志（可选） |

## 15. 推荐测试主线

建议至少覆盖以下主线测试：

1. 页面恢复上次状态
2. 新建会话 → 输入草稿 → 切换会话 → 草稿恢复
3. 发送消息 → 生成计划卡 → 图上出现对象
4. 点击计划卡 → 图表跳转高亮
5. 点击图上对象 → 回跳消息
6. 回放推进 → TP/SL/过期状态正确变化
7. 标记管理器按会话/消息/对象筛选稳定生效
8. 切换 AI 时会话摘要保留成功
9. 切换 AI 后继续追问，回复能接续前文
10. 错误场景下草稿与会话不丢失

## 16. 给实现者的最终建议

如果你要快速实现一个稳定版本，建议按以下顺序开发：

### 第一阶段
- 页面恢复
- 会话切换
- 草稿保存
- 发送消息
- 渲染消息

### 第二阶段
- 计划卡
- 标记对象
- 图表联动
- 标记管理器

### 第三阶段
- 生命周期推进
- AI 切换
- SessionMemory
- handoff packet

这样最稳，也最容易测试。
